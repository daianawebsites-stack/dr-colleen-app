Motion & easing

Four duration steps — each has a specific voice. Fast = “tight, responsive”; standard = most UI feedback; moderate = overlay fade; slow = content reveal & background wash. All use the same easing curve.

TokenDurationUse cases
fast0.15sButton hover, tap scale, save CTA state change
standard0.2sColor, opacity, border transitions · default
moderate0.25sModal fade, drawer slide, view toggle
slow0.3sAccordion expand, bottom-sheet slide, bar fill
easing.smooth — cubic-bezier(0.4, 0, 0.2, 1) — is the one-and-only easing curve. Applied to the nav-pill width animation, accordion rotation, carousel translate, and quiz slide.

07 · Patterns

Rules & behaviors

Ten cross-cutting rules that govern how the pieces compose. These are not style tokens — they’re the logic that prevents visual or semantic drift as new screens are added.

7.1 · Severity resolution (3-tier fallback)
getDynamicSeverity · L271
Every rating (1–5) resolves to one of three semantic colors. The resolver first checks symptom-specific overrides (L245), then falls back to the shared scale: 1–2 green / 3 amber / 4–5 rose. Every severity-rendering component uses this — never hardcode.
1 · Mild 2 · Mild 3 · Moderate 4 · Severe 5 · Severe
7.2 · Resources-DB as source of truth
getResourceChipLabel · L106
Every ResourceLinkChip pulls its label, category, and target URL from a single resources DB. Chip labels are never authored per screen — the DB is the single source, and getResourceChipLabel(id) returns the display text so that rebranding or copy edits stay in one place.
7.3 · Per-screen chip dedup
pickUniqueResourceLinks · L128
Within a single screen render, each resource chip appears at most once. The picker walks cards in order and skips already-used IDs, so a popular article doesn’t repeat across symptom cards. Empty results are silently dropped.
7.4 · Two module-body rhythms
moduleBodyPadding vs moduleContentPadding
Every module body picks exactly one of the two padding patterns from Foundation § Layout. Pattern A (Card): multiple peer cards separated by hairlines — 20/32/20 rhythm. Pattern B (Content): single block of freeform content, 20/20. Never mix inside a single module.
7.5 · Dismissible advice state
root App component
Dismissed advice IDs are held in root-level App state (not inside the card). When the user taps X on a Nutritional Advice card, the ID is appended to dismissedAdvice and the card is filtered out next render. State resets on logout — never persisted, intentionally lightweight.
7.6 · Drag-to-reorder (useModuleOrder)
L6722
Module order is managed by the useModuleOrder hook, which takes an initial array and returns {order, onDragStart, onDragOver, onDrop, dragging}. Pass these to each CollapsibleModule. Draggable modules are not used in Health Report — Section wraps them without drag.
7.7 · Orphan hairline (isLast)
shared by SymptomBar, FreqRow, InfoRow, card rows
Every row component accepts an isLast prop that suppresses the bottom hairline of the final row. This avoids the trailing-divider “orphan” look where a hairline sits with nothing after it.
7.8 · Severity-tint semantic reuse
SEVERITY_BG serves FreqRow, advice, calendar
The three severity tint colors (#fff0f3 / #fff5ec / #f0faf0) are deliberately reused for food-frequency scoring, advice status, and future alert surfaces. Context (not color) disambiguates — amber in a symptom means “moderate severity”, amber in food means “too frequent”.
7.9 · Demo vs production persistence
TRACKER_PERSISTENCE_ENABLED · L4770
Tracker entries write to local state only — no backend call — when TRACKER_PERSISTENCE_ENABLED = false. This is the default for the prototype so walkthroughs stay clean. Flip to true when wiring to the real backend. The flag never changes any visuals.
7.10 · Screen transitions
fadeInPage · L4387 · chat always mounted
Chat is always mounted — it never animates, so the user returns to the exact scroll position. Every other screen enters with fadeInPage 0.3s easing.smooth. No slide, no scale, no cross-fade — only opacity. The drawer uses its own slideIn/Out keyframes.
